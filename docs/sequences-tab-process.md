# Sequences Tab Process Document

## Purpose

This document explains how the Sequences tab works internally, which files participate in the flow, and how data moves from the UI to storage and back through execution.

## High-Level Architecture

The feature is split into four layers:

1. UI layer
2. API route layer
3. store/service layer
4. database tables

Main implementation files:

- `src/app/(app)/sequences/SequencesClient.tsx`
- `src/app/api/sequences/route.ts`
- `src/app/api/sequences/[sequenceId]/route.ts`
- `src/app/api/sequences/[sequenceId]/enroll/route.ts`
- `src/app/api/sequences/[sequenceId]/enrollments/[enrollmentId]/route.ts`
- `src/app/api/sequences/run-due/route.ts`
- `src/app/(app)/tasks/TasksClient.tsx`
- `src/app/api/tasks/route.ts`
- `src/app/api/tasks/[taskId]/route.ts`
- `src/lib/sequences-store.ts`

## Data Storage Model

The Sequences feature uses custom tables created in:

- `prisma/migrations/20260409_add_sequence_backend_tables/migration.sql`

The same table creation logic also exists in `ensureSequencesTables()` inside `src/lib/sequences-store.ts`, so the feature can self-create required tables at runtime.

### Tables

#### `app_sequences`

Stores sequence definitions:

- sequence metadata
- status
- settings JSON
- steps JSON

#### `app_sequence_enrollments`

Stores one row per contact enrolled in a sequence:

- current step index
- next run timestamp
- paused/completed/stopped state
- finish reason
- last activity time

Unique constraint:

- one contact can only be enrolled once per sequence

#### `app_sequence_events`

Stores the sequence event log used by the Recent Activity panel.

#### `app_tasks`

Stores manual follow-up work generated by sequence steps such as:

- manual emails
- phone calls
- LinkedIn touches
- generic tasks

## Core Types

Defined in `src/lib/sequences-store.ts` and mirrored in the client:

- `SequenceStatus`: `draft | active | paused | archived`
- `SequenceStepType`: `automatic_email | manual_email | phone_call | task | linkedin_task`
- `EnrollmentStatus`: `active | paused | completed | stopped`

## Request And Execution Flow

### 1. Load The Sequences Page

Route:

- `/sequences`

Rendered by:

- `src/app/(app)/sequences/page.tsx`
- `src/app/(app)/sequences/SequencesClient.tsx`

The client loads data with SWR:

- sequence list from `GET /api/sequences`
- selected sequence detail from `GET /api/sequences/:sequenceId`
- contact options from `GET /api/contacts`

### 2. List Sequences

Request:

- `GET /api/sequences?page=&limit=&search=&status=`

Flow:

1. Route handler validates the current user.
2. It calls `listSequences(orgId, filters)`.
3. `listSequences()` ensures tables exist.
4. The store queries `app_sequences`.
5. It joins enrollment counts per sequence.
6. It returns paginated rows plus summary metrics.

Important details:

- active sequences are sorted first, then paused, then draft, then archived
- summary cards are calculated separately from the paginated list

### 3. Get Sequence Detail

Request:

- `GET /api/sequences/:sequenceId`

Flow:

1. Route handler checks auth.
2. It calls `getSequenceDetail(orgId, sequenceId)`.
3. The store loads:
   - the sequence row
   - enrollments for that sequence
   - the 12 most recent events
4. The response is returned to the client.

This powers:

- the builder
- enrolled contacts
- recent activity

### 4. Create A Sequence

Request:

- `POST /api/sequences`

Flow:

1. The client posts the current draft.
2. `createSequence()` sanitizes and normalizes the payload.
3. The store inserts a row into `app_sequences`.
4. A `sequence_created` event is recorded.
5. Full sequence detail is returned.

Normalization behavior:

- name is required
- description is trimmed
- settings are normalized with defaults
- steps are normalized, deduplicated by `id`, and defaulted if missing
- delay days are clamped to `0..30`

### 5. Update A Sequence

Request:

- `PATCH /api/sequences/:sequenceId`

Flow:

1. The client submits the current draft.
2. `updateSequence()` loads the existing sequence.
3. The store normalizes the new values.
4. The sequence row is updated.
5. A `sequence_updated` event is recorded.
6. If the new status is `active`, active enrollments with no `next_run_at` are scheduled immediately.
7. Full detail is returned.

## Enrollment Process

### 6. Enroll Contacts

Request:

- `POST /api/sequences/:sequenceId/enroll`

Flow:

1. The client sends an array of selected contact ids.
2. `enrollContactsInSequence()` removes duplicates and validates the contacts belong to the org.
3. For each valid contact, the store inserts a row into `app_sequence_enrollments`.
4. If the sequence is active, `next_run_at` is set to `NOW()`.
5. A `contact_enrolled` event is recorded for new inserts.
6. Updated sequence detail is returned.

Important details:

- duplicate contact enrollment is prevented by a unique constraint
- if the sequence is not active, `next_run_at` is stored as `NULL`

### 7. Change Enrollment Status

Request:

- `PATCH /api/sequences/:sequenceId/enrollments/:enrollmentId`

Supported actions:

- `pause`
- `resume`
- `complete`
- `stop`

Behavior:

- `pause` sets status to `paused`, clears `next_run_at`, and stores `paused_until = now + 24h`
- `resume` sets status to `active` and `next_run_at = now`
- `complete` sets status to `completed`
- `stop` sets status to `stopped`

Each action records a sequence event.

Important note:

- `paused_until` is informational in the current implementation; nothing auto-resumes when that time arrives

## Sequence Runner Process

### 8. Run Due Steps

Request:

- `POST /api/sequences/run-due`

Store method:

- `runDueSequenceSteps(orgId, actorUserId)`

Selection rules:

- sequence status must be `active`
- enrollment status must be `active`
- `next_run_at` must be set
- `next_run_at <= NOW()`
- max 100 enrollments per run

### 9. Runner Logic

For each due enrollment:

1. Load the current step using `current_step_index`.
2. If the step is missing, complete the sequence.
3. If the step is `automatic_email`:
   - send email through SMTP, or simulate if SMTP is not configured
   - record an `email_sent` or `email_simulated` event
   - advance to the next step
4. If the step is manual/task-based:
   - check whether an open task already exists for this enrollment and step index
   - create a task if none exists
   - record `task_created`
   - clear `next_run_at`
   - wait for the task to be completed

If an automatic email step has no contact email:

- the step is skipped
- `email_skipped` is recorded
- the enrollment still advances

### 10. Advancing To The Next Step

Method:

- `advanceEnrollmentToNextStep()`

Behavior:

- if there is another step:
  - increment `current_step_index`
  - schedule `next_run_at = now + delayDays`
  - record `step_scheduled`
- if there is no next step:
  - mark enrollment `completed`
  - set `finished_reason = sequence_finished`
  - clear `next_run_at`
  - record `sequence_completed`

## Manual Task Completion Process

### 11. Task Creation

Manual steps create rows in `app_tasks` with:

- `sequence_id`
- `enrollment_id`
- `contact_id`
- `metadata.stepIndex`
- `metadata.stepType`
- `metadata.stepTitle`

This metadata links the task back to the sequence step that generated it.

### 12. Completing A Task

Request:

- `PATCH /api/tasks/:taskId`

Flow:

1. The Tasks page sends `complete` or `reopen`.
2. `updateTaskCompletion()` updates task status.
3. If the action is `complete` and the task belongs to a sequence enrollment:
   - metadata is read
   - current sequence detail is loaded
   - the matching enrollment is checked
   - if the enrollment is still on the same step index, a `task_completed` event is recorded
   - `advanceEnrollmentToNextStep()` is called

Important details:

- reopening a task does not roll the sequence back
- task completion only advances the enrollment if the task still matches the enrollment's current step

## UI State Flow

The client in `SequencesClient.tsx` keeps local state for:

- search and filters
- selected sequence
- create mode vs edit mode
- editable draft values
- selected contacts for enrollment
- notices and errors

The page uses SWR to refresh:

- list data after save, enroll, and run-due actions
- detail data after selection changes and mutation results

## Sequence Of User Actions

Typical happy-path flow:

1. User opens `/sequences`.
2. User creates a sequence and saves it.
3. User activates the sequence.
4. User enrolls contacts.
5. Active enrollments get `next_run_at`.
6. User clicks `Run Due Steps`.
7. Automatic steps send or simulate emails.
8. Manual steps generate tasks in `/tasks`.
9. User completes the task in the Tasks page.
10. The sequence advances and schedules the next step.
11. Final step completion marks the enrollment as completed.

## Known Implementation Notes

### Stored But Not Enforced Yet

These values are stored and displayed but not used in runner scheduling logic:

- `settings.timezone`
- `settings.scheduleName`
- `settings.useLocalTimezone`

### Email Sending Mode

Automatic email sending depends on:

- `SMTP_HOST`
- `SMTP_PORT`
- `SMTP_USER`
- `SMTP_PASS`
- `SMTP_FROM`

Without valid values, the system falls back to simulation mode.

### Raw SQL Ownership

The sequence feature is mostly implemented with raw SQL in `src/lib/sequences-store.ts` instead of Prisma models. If this area is extended, changes usually need to be made in both:

- the migration
- the store queries and inserts

### Runtime Table Creation

`ensureSequencesTables()` creates the required tables if they do not exist. That makes the module resilient, but it also means schema behavior is partly owned in application code instead of Prisma schema models.

## Recommended Operational Usage

Until an automatic scheduler exists, the safest operating model is:

1. keep sequences in `draft` while editing
2. activate only when ready
3. run `Run Due Steps` on a regular cadence
4. monitor the Tasks page for manual steps
5. complete generated tasks promptly so enrollments can continue
